# Fix layout example save — race condition + stale-board guard

## What changed

- **`renderLayoutExample`**: moved `_leLoadedSlug = slug` and `_leBoard = board` **inside** the `send()` function, so they are only set *after* the `loadBoard` postMessage has actually been dispatched to the iframe. Previously they were set before `send()`, meaning a user clicking Save between those two lines could capture the *previous* component's board and save it under the wrong slug.

- **`leSaveLayout`**: added an early-exit guard — if `_leLoadedSlug !== selectedSlug` the save is aborted with a clear message ("Layout not ready yet — please wait a moment and try again"). This prevents saving a stale board from a component the iframe was showing before the current one finished loading.

- **`leSaveLayout`**: after a successful PUT to the server, `_leBoard` is now updated with the just-saved board. This keeps the in-memory cache in sync so the early-return in `renderLayoutExample` serves the fresh saved state rather than the pre-edit snapshot.

## Files
- `frontend/static/index.html`

## Notes
The save API endpoint and the board-capture postMessage flow were correct all along; the bug was a timing/ordering issue where state variables were set too early relative to the actual iframe communication.
